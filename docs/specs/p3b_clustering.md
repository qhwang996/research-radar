# P3b Theme Model + Clustering Pipeline - Implementation Spec

## 1. Goal

将高相关论文按研究子领域聚类，生成 Theme（研究主题）记录。这是智能分析层的第二步，把 P3a 产出的 ~200 篇结构化 L2 分析组织成「研究前沿地图」。

例如，200 篇高相关安全论文可能聚类成 10-15 个 Theme：
- "Web Application Fuzzing"（15 篇）
- "Server-Side Vulnerability Detection"（8 篇）
- "JavaScript/Browser Security"（12 篇）
- "Binary Analysis for Vulnerability Discovery"（20 篇）
- ...

完成后，后续的趋势分析（P3c）和方向综合（P3d）都基于 Theme 工作。

---

## 2. Dependencies

- **P3a DeepAnalysisPipeline** — 需要 `summary_l2` JSON（research_problem, methodology, related_concepts）
- **LLMClient** — STANDARD tier（运行时映射到 haiku），cache + retry
- **ProfileRepository** — 用户研究上下文

无新增外部依赖。

class ThemeStatus(str, Enum):
    """Lifecycle states for a research theme."""

    CANDIDATE = "candidate"   # 系统生成，待确认
    CORE = "core"             # 用户确认的核心主题

### 3.2 新增模型

新建 `src/models/theme.py`：

```python
class Theme(TimestampedModel, Base):
    """Research sub-area cluster generated by the clustering pipeline."""

    __tablename__ = "themes"

    theme_id: Mapped[str]              # String(36), unique, indexed, UUID
    name: Mapped[str]                  # String(200), 主题名称
    description: Mapped[str | None]    # Text, 2-3 句话描述
    keywords: Mapped[list[str]]        # JSON, 代表性关键词
    artifact_ids: Mapped[list[int]]    # JSON, 关联论文的 artifact.id
    artifact_count: Mapped[int]        # Integer, 冗余计数
    paper_count_by_year: Mapped[dict]  # JSON, {"2022": 5, "2023": 12}
    methodology_tags: Mapped[list[str]]# JSON, 常见研究方法（P3c 填充）
    open_questions: Mapped[list[str]]  # JSON, 开放问题（P3c 填充）
    trend_direction: Mapped[str | None]# String(20), growing/stable/declining（P3c 填充）
    status: Mapped[ThemeStatus]        # Enum, 默认 CANDIDATE
    generation_version: Mapped[str]    # String(50), 如 "v1"
    week_id: Mapped[str]               # String(10), 如 "2026-W11"
```

**字段说明**：
- `theme_id` 使用 UUID4，与 Artifact 的 `canonical_id` 模式一致
- `artifact_ids` 存放 `artifact.id`（整数主键），不是 `canonical_id`
- `paper_count_by_year` 由聚类 pipeline 计算并填入
- `methodology_tags` / `open_questions` / `trend_direction` 初始为空/None，由 P3c TrendAnalysisPipeline 填充
- `status` 默认 `CANDIDATE`，用户通过 feedback 确认后变为 `CORE`
- `week_id` 格式为 ISO week，如 `"2026-W11"`

**字段类型参照** `Artifact` 模型的做法：
- `list[str]` 用 `MutableList.as_mutable(JSON)`
- `dict` 用 `MutableDict.as_mutable(JSON)`
- `ThemeStatus` 用 `SAEnum(ThemeStatus, values_callable=_enum_values, native_enum=False)`

## 4. 聚类 Pipeline

### 4.1 ClusteringPipeline

新建 `src/pipelines/clustering.py`：

```python
class ClusteringPipeline(BasePipeline):
    """Group high-relevance papers into research themes via LLM clustering."""
```

**这不是逐条 pipeline**，而是批量操作。与 enrichment/deep_analysis 的模式不同：
- 输入：一批论文的压缩表示
- 输出：一组 Theme 记录
- 不用 ThreadPoolExecutor（批次间有依赖关系：merge 需要所有批次的结果）

### 4.2 构造函数

```python
def __init__(
    self,
    *,
    session_factory: sessionmaker[Session] | None = None,
    llm_client: LLMClient | Any | None = None,
    prompt_template_path: Path | None = None,
    cluster_version: str = "v1",
    min_relevance: float = 0.6,
    batch_size: int = 35,
) -> None:
```

| 参数 | 默认值 | 说明 |
|------|--------|------|
| cluster_version | `"v1"` | 版本号，全量重聚类时用于清理旧版 CANDIDATE |
| min_relevance | `0.6` | 筛选阈值 |
| batch_size | `35` | 每批发给 LLM 的论文数 |

### 4.3 process() 方法

```python
def process(self, input_data: Any = None) -> list[Theme]:
```

`input_data` 支持：
- `None` → 全量聚类（默认）
- `"incremental"` → 增量模式（分类新论文到已有 Theme）

**全量聚类流程**：

1. **收集**：查询所有 `status==ACTIVE, source_type==PAPERS, relevance_score>=0.6, summary_l2 IS NOT NULL` 的 artifact
2. **压缩表示**：每篇提取 `{id, title, research_problem, methodology, related_concepts}`（从 `summary_l2` JSON 解析）
3. **计算 week_id**：当前 ISO week
4. **分批聚类**：
   - 将论文按 batch_size 分批
   - 每批发一次 LLM 调用，要求返回 `[{cluster_label, description, paper_ids}, ...]`
   - cache_key: `cluster_batch_{cluster_version}_{batch_index}_{sha256(sorted_ids_in_batch)}`
5. **合并**：
   - 收集所有批次的聚类结果
   - 发一次 merge LLM 调用，输入所有 cluster_label + description，要求合并语义相似的聚类
   - cache_key: `cluster_merge_{cluster_version}_{sha256(sorted_all_labels)}`
6. **持久化**：
   - 先删除同 `cluster_version` 的旧 CANDIDATE Theme（`delete_candidates_by_version`）
   - 为每个合并后的聚类创建 Theme 记录
   - 计算 `paper_count_by_year` 和 `artifact_count`
   - 保留已有的 CORE Theme 不动

**增量模式流程**（简化版，首次实现可以不做，标注 `# TODO: incremental`）：

1. 查询没有被任何 Theme 引用的新高相关论文
2. 对每篇新论文，发 LLM 调用问「应该归入哪个已有 Theme，还是需要新建」
3. 更新对应 Theme 的 artifact_ids
4. 累积 >10 篇无法分类的论文时，log 一条建议全量重聚类的 warning

### 4.4 Prompt 设计

新建 `prompts/cluster_papers.md`（分批聚类用）：

```
你是一位安全研究领域的论文聚类专家。

以下是一批安全领域的研究论文摘要。请将它们按研究子领域聚类。

用户研究方向：{{research_area}}

论文列表：
{{paper_list}}

请返回一个 JSON 数组，每个元素表示一个聚类：

[
  {
    "cluster_label": "聚类名称（英文短语，如 Web Application Fuzzing）",
    "description": "这个研究子领域的 2-3 句话描述",
    "paper_ids": [论文ID列表],
    "keywords": ["关键词1", "关键词2", "关键词3"]
  }
]

要求：
- 每个论文必须且只能属于一个聚类
- 聚类名称应反映研究问题，不是泛化描述
- 避免太大的聚类（单个聚类不应超过论文总数的 40%）
- 避免太小的聚类（少于 2 篇的考虑合并到相近聚类）
- 聚类基于研究问题和方法论的相似性，不仅是关键词重叠
- 请只返回 JSON，不要额外解释
```

其中 `{{paper_list}}` 格式：

```
[ID=42] 标题: XXX
  研究问题: YYY
  方法: ZZZ
  相关概念: A, B, C

[ID=57] 标题: XXX
  ...
```

**Merge prompt**（合并用，可以内嵌在代码中或单独模板）：

```
以下是从多个批次产生的论文聚类标签。请合并语义相似的聚类。

聚类列表：
{{cluster_list}}

请返回合并后的聚类映射 JSON：

[
  {
    "final_label": "合并后的聚类名称",
    "description": "合并后的描述",
    "merged_from": ["原始标签1", "原始标签2"],
    "keywords": ["关键词1", "关键词2"]
  }
]

要求：
- 语义相似的聚类合并为一个（如 "Web Fuzzing" 和 "Browser Fuzzing" 可以合并为 "Web/Browser Fuzzing"）
- 不相似的聚类保持独立
- 没有需要合并的就原样返回（merged_from 只包含自身标签）
- 请只返回 JSON，不要额外解释
```

### 4.5 LLM 配置

| 调用 | Tier | max_tokens | temperature | 说明 |
|------|------|-----------|-------------|------|
| 分批聚类 | STANDARD | 2000 | 0.3 | 每批 30-40 篇 |
| Merge | STANDARD | 1500 | 0.2 | 合并标签，运行时都是 haiku |

> 原设计 merge 用 PREMIUM，但因为只有 haiku 可用，改为 STANDARD。代码中保留 tier 参数便于后续升级。

### 4.6 paper_count_by_year 计算

在持久化 Theme 时，根据 artifact_ids 查询对应 artifact 的 year 字段，统计每年论文数：

```python
def _compute_year_counts(self, artifact_ids: list[int], session: Session) -> dict[str, int]:
    repo = ArtifactRepository(session)
    counts: dict[str, int] = {}
    for aid in artifact_ids:
        artifact = repo.get_by_id(aid)
        if artifact and artifact.year:
            key = str(artifact.year)
            counts[key] = counts.get(key, 0) + 1
    return counts
```

或更高效的批量查询（推荐）。

### 4.7 JSON 解析

LLM 返回 JSON 数组（不是对象），解析方式：
- `_strip_code_fences` + `json.loads`
- 验证是 `list[dict]`
- 每个 dict 必须有 `cluster_label`（str）和 `paper_ids`（list[int]）
- `description` 和 `keywords` 降级为空字符串/空列表
- 如果某个 paper_id 不在输入列表中，忽略并 warn

## 11. 关键注意事项

### 11.1 LLM 返回的是 JSON 数组

与之前的 pipeline 不同（返回 JSON 对象），聚类 prompt 返回的是 `[{...}, {...}]`。解析时注意 `json.loads` 后验证是 `list`。

### 11.2 paper_ids 对齐

LLM 可能返回不存在于输入的 paper_id，或遗漏某些论文。需要：
- 忽略不在输入列表中的 ID
- 将未被分配的论文收集到一个 "Uncategorized" Theme 或 log warning
- 验证所有输入论文都被分配到某个 Theme

### 11.3 batch 分割的确定性

分批时按 `artifact.id` 排序后顺序分割，保证 cache_key 稳定。

### 11.4 CORE Theme 保护

全量重聚类时：
1. 先查询所有 CORE Theme，记录它们的 artifact_ids
2. 从聚类输入中排除这些 artifact_ids（CORE Theme 的论文不重新分配）
3. 只对剩余论文做聚类
4. 新旧 CANDIDATE Theme 替换，CORE Theme 保持不动

如果首次实现觉得太复杂，可以简化为：不排除 CORE 论文，但重聚类后保留 CORE Theme 记录不删除。新聚类可能与 CORE Theme 有论文重叠，这是可以接受的。
