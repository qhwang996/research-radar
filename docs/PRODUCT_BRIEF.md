# PRODUCT_BRIEF

## 1. 产品一句话

Research Radar 现在的定位是：

> 面向安全科研的前沿情报收集、分层消费与周期总结系统。

它服务的不是“单次选题”，而是博士研究过程中的长期情报工作流。

## 2. 目标用户

单用户、时间极少、需要长期跟踪安全学术与工业前沿的安全科研工作者。

当前默认用户画像：
- 以安全科研为主
- 学术产出优先
- 需要同时跟踪工业痛点，但不能被工业信息噪声淹没
- 不愿长期手动维护系统

## 3. 要解决的核心问题

### 3.1 日常问题
“今天有什么值得我注意的内容？哪些必须点开原文？哪些看一句话就够？”

### 3.2 周度问题
“这周哪些主题在持续升温？学术与工业之间有哪些值得关注的张力？”

### 3.3 月度问题
“过去一个月，哪些信号不是噪音而是稳定趋势？哪些阅读值得转成研究积累？”

### 3.4 季度问题
“过去一个季度，哪些方向值得真正进入选题/子课题池？哪些 sources 值得保留或移除？”

## 4. 输出层级

系统输出分 5 层：

1. **Read original**
   - 用户亲自读原文
   - 只给最少量、高价值、不可替代的条目

2. **Detailed summary**
   - 用户不立即读原文，但愿意花几分钟看详细摘要

3. **One-line summary**
   - 用户只需要知道“发生了什么”

4. **Delegate to agent**
   - 内容高价值但难读、长、抽象、或不值得用户立即投入原文阅读时间
   - 交给通用 agent 代读并产出结构化回报

5. **Archive**
   - 已读、重复、低价值、长期不相关、或确认噪声的内容

## 5. 轨道原则

### Academic track
以论文为主：
- T1 顶会
- T2 arXiv
- 后续可选 SE 安全相关会议轨

### Industry track
以工业需求和现实痛点为主：
- T3 研究博客
- Advisories（后续）
- T4 个人博客 / 公众号（用户手选后）

### 不可混排规则
以下内容不能直接放进同一个 Top-N：
- 论文与博客
- 学术证据与工业痛点
- 学术重要性排序与工业紧迫性排序
- “方向发现”结论与底层单条情报

## 6. 研究方向发现的新位置

“研究方向发现”不删除，但改为上层输出：

- Daily：不做方向发现，只做情报消费
- Weekly：给出初步 gap / direction shortlist
- Monthly：给出稳定方向候选与阅读积累总结
- Quarterly：给出真正值得进入研究路线图的方向

## 7. 非目标

当前阶段明确不做：

- 前端产品
- 多用户权限系统
- 实时推送系统
- 全自动论文精读代理闭环
- 高度复杂的在线个性化学习模型
- 大规模 source 扩张而不先解决 triage / report

## 8. 成功标准

### 必须满足
- 每天自动产出可读日报
- 学术与工业严格分开
- 用户无需频繁手动筛选
- 周/月/季度都有明确、稳定输出
- delegate 通道能处理“高价值但难读”的条目
- feedback 可以回流

### 最终理想状态
用户只需要：
- 每天看日报 5–10 分钟
- 每周看一次周报
- 每月/每季度看一次综合总结
- 很少介入系统管理

## 9. 实施原则

- 保持现有 CLI / pipeline / reporting 骨架
- 不从零重构
- 优先补文档、策略和局部实现
- 每次改动都必须可重跑、可回退、可测试
